# EPUB Structured Batch Translation Design

This document defines a proposed improvement to the existing EPUB translation pipeline in `lexora-ai`.

Status: **Implemented and closed** — **LAI-T-032**–**LAI-T-035** **Done** in [todo-list.md](todo-list.md) (2026-04-15). Design + plan: [epub-structured-batch-translation-plan.md](epub-structured-batch-translation-plan.md). Operators should still validate real-provider JSON on representative EPUBs.

## 1. Objective

Improve the current translation logic by reducing token overhead and request latency while preserving EPUB structure deterministically.

Core goals:

- Preserve DOM invariants and output mapping guarantees.
- Reduce provider round-trips by batching multiple translation items per request.
- Keep translation behavior compatible with current cache and run-report contracts.
- Support OpenAI GPT API and Azure AI Foundry GPT-4 as first providers.

## 2. Current Baseline (Already Implemented)

Current canonical logic is documented in `docs/translation-logic.md`:

- EPUB text nodes are collected with DOM-aware parsing.
- Long text is sentence-safe chunked.
- Uncached chunks are translated through provider batch calls.
- Optional merge mode packs chunks with markers and reassembles by marker parsing.

The baseline works, but very granular node/chunk processing can still cause:

- High prompt overhead per API call.
- Large number of requests on long books.
- Longer end-to-end run time.

## 3. Proposed Improvement

Add a structure-preserving JSON batching mode for EPUB translation.

Instead of sending one small chunk per provider request, send multiple items in one request using explicit IDs. Each item is a translation unit derived from DOM blocks (for example paragraph/list item/heading), with optional local context.

The model returns a strict JSON object containing only translated outputs mapped by the same IDs.

## 4. Translation Unit Contract

Each unit is mapped to a deterministic source span in the parsed DOM.

Required fields:

- `id`: stable run-local unique ID for mapping.
- `text`: source text to translate.

Optional fields:

- `type`: semantic DOM class (`paragraph`, `heading`, `list_item`, ...).
- `context_before`: neighboring text snippet (not for output replacement).
- `context_after`: neighboring text snippet (not for output replacement).

### Request Schema (Conceptual)

```json
{
  "source_lang": "en",
  "target_lang": "vi",
  "batch_id": "doc003_batch014",
  "items": [
    {
      "id": "doc003:p0001",
      "type": "paragraph",
      "text": "Hello world",
      "context_before": "",
      "context_after": "How are you?"
    },
    {
      "id": "doc003:p0002",
      "type": "paragraph",
      "text": "How are you?",
      "context_before": "Hello world",
      "context_after": ""
    }
  ]
}
```

### Response Schema (Conceptual)

```json
{
  "items": [
    { "id": "doc003:p0001", "translated_text": "Xin chao the gioi" },
    { "id": "doc003:p0002", "translated_text": "Ban co khoe khong?" }
  ]
}
```

## 5. Strict Validation Rules

A response is accepted only when all conditions pass:

1. Valid JSON object.
2. Root has `items` array.
3. Output item count equals input item count.
4. IDs exactly match input ID set (no missing/extra/duplicate IDs).
5. Every item has `translated_text` string.
6. Non-empty source requires non-empty translation.

If validation fails:

1. One retry with JSON repair instruction.
2. If still invalid, split batch and retry smaller batches.
3. Final fallback to current stable path (existing chunk-level flow).

## 6. Batching Strategy

Batching must be token-budget based, not fixed count.

Design guidance:

- Compute estimated input tokens for prompt + serialized JSON payload.
- Reserve output token budget proportional to input text.
- Keep a safety margin to avoid context overflow.
- Build batches greedily until budget threshold.

Example policy (initial):

- Target input budget: conservative fraction of model context window.
- Reserve output budget: enough for full translated JSON response.
- Tune by observed latency, JSON validity rate, and truncation risk.

## 7. Provider Scope (Phase 1)

First-class providers:

- OpenAI GPT API (JSON output mode / schema-constrained output where available).
- Azure AI Foundry GPT-4 deployment (same contract, adapter-level differences only).

Provider adapters should expose one unified internal operation:

- `translate_structured_batch(request) -> structured_response`

Core pipeline must stay provider-agnostic and stateless.

## 8. Structure Preservation Guarantees

This proposal must keep existing EPUB invariants:

- Node/chunk replacement mapping remains deterministic by `id`.
- DOM structure is not regenerated by model output.
- Non-translatable tags remain skipped (`script`, `style`, `code`, `pre`, etc.).
- Spine, TOC, metadata, and link integrity remain unchanged by translation layer.

## 9. Cache Compatibility

Keep behavior-aware cache strategy from current pipeline.

For structured mode, cache keys should include:

- Source text hash (normalized and raw strategy as defined by implementation).
- Source/target language.
- Provider + model identity.
- Instruction/glossary fingerprints.
- Chunking/structured-batch pipeline version fingerprint.

Cache lookup happens at per-item level before provider batching, then uncached items are packed into batches.

## 10. Observability and Contract Impact

This change should preserve and extend current reporting expectations:

- Existing run report fields remain valid.
- `translation_summary` should include structured-batch metrics when mode is enabled.

Suggested new summary fields:

- `structured_batch_enabled`
- `structured_batches_total`
- `structured_items_total`
- `structured_validation_failures`
- `structured_fallback_batches`

## 11. Rollout Strategy

Recommended phased rollout:

1. Add feature flag for structured-batch mode (default off).
2. Enable in controlled test runs with Golden EPUB suite.
3. Compare quality/cost/time against current merge/chunk path.
4. Promote to default only after stability thresholds are met.

## 12. Risks and Mitigations

Risk: Model returns malformed JSON.
- Mitigation: strict validator, repair retry, batch split fallback.

Risk: Token budget misestimation causes truncation.
- Mitigation: conservative tokenizer budget and safety margin.

Risk: Context quality regression due to larger batch boundaries.
- Mitigation: local neighbor fields and quality regression tests on Golden EPUB fixtures.

Risk: Complex inline-rich blocks lose nuance.
- Mitigation: hybrid policy to route fragile blocks to smaller units.
